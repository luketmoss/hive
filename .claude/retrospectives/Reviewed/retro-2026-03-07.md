# Pipeline Optimization Audit — 2026-03-07

## Purpose

Comprehensive analysis of the Hive agent pipeline's efficiency, accuracy, and consistency. Based on 3 prior retrospectives, 20 merged PRs, and comment/lifecycle data from all completed GitHub issues. Goal: identify actionable changes to skills, CLAUDE.md, and tooling to reduce user intervention, eliminate repeated failures, and cut pipeline execution time.

## Executive Summary

The pipeline's design is sound — PM, UX, Dev, QA, and Review stages catch real issues and produce quality output. But **execution consistency is poor**: the orchestrator stops when it shouldn't, agents use broken commands (`jq`), and quality gates miss entire categories of bugs (migration/day-zero). The biggest wins come from:

1. **Fixing skill handoff text** that causes the model to stop between stages (70 min wasted on #77)
2. **Fixing `jq` in all Board Movement Helpers** (fails every time on Windows)
3. **Adding migration ACs to PM** (4 post-merge bugs on #41)
4. **Adding empty-state testing to QA** (all 4 #41 bugs would have been caught)

## Finding 1: Skill Handoff Text Causes Pipeline Stops

**Severity: P0 — Root cause of most user interventions**

### Evidence

From the #77 analysis (126 min wall clock, 15.8 min active):

| After Stage | Idle Time | User Nudge Required |
|-------------|-----------|---------------------|
| PM | 16.1 min | "ready" |
| UX | 30.0 min | "ok" |
| PM+UX | 0.7 min | "ok" |
| Dev+Verify | 9.4 min | "ok" |
| QA | 14.1 min | "ok" |
| **Total** | **70.1 min** | **5 nudges** |

Agent utilization was **12.7%**. The pipeline was designed to be autonomous but stopped after every skill invocation.

### Root Cause

Every skill ends with conversational handoff text that the model treats as a conversation-ending turn:

| Skill | Handoff Text |
|-------|-------------|
| PM | "Issue #N refined and moved to Ready. When you're ready to implement, run `/dev #N`." |
| Dev | "PR #X opened for issue #N, moved to Testing. When you're ready to verify, run `/qa #N`." |
| QA | "QA passed for issue #N. Run `/review #N` for code review." |
| Review | "Code review passed for PR #X. Ready for user approval and merge." |
| UX | "UX review complete. Findings posted to #N. Address critical items before shipping." |
| Idea | "Issue #N created in To Do. When you're ready to refine it, run `/pm #N`." |

Two problems:
1. The text reads as a **final message** — the model emits `end_turn`
2. The text says "when **you're** ready, run `/dev`" — actively telling the **user** to invoke the next step, contradicting CLAUDE.md which says the orchestrator should continue autonomously

### Proposed Fix

Replace all handoff sections with a neutral completion signal. Remove language that addresses the user or suggests next steps — the orchestrator already knows what to do.

**Before:**
```markdown
## Handoff
When complete, tell the user:
> Issue #N refined and moved to Ready. When you're ready to implement, run `/dev #N`.
```

**After:**
```markdown
## Handoff
When complete, output a brief status line:
> PM complete — Issue #N: <AC count> ACs defined, moved to Ready.
Do NOT suggest next steps or address the user. The orchestrator will decide what happens next.
```

### Files to Change
- `.claude/skills/pm/SKILL.md` — Handoff section
- `.claude/skills/dev/SKILL.md` — Handoff section
- `.claude/skills/qa/SKILL.md` — Handoff section (3 verdicts)
- `.claude/skills/review/SKILL.md` — Handoff section (2 verdicts)
- `.claude/skills/ux/SKILL.md` — Handoff section
- `.claude/skills/idea/SKILL.md` — Handoff section

## Finding 2: `jq` in Every Board Movement Helper (Breaks on Windows)

**Severity: P0 — Causes command failures every pipeline run**

### Evidence

User-reported `jq: command not found` error. The Board Movement Helper in all 6 skill files uses:

```bash
--id "$(gh project item-list 2 --owner luketmoss --format json | jq -r '.items[] | select(...) | .id')"
```

CLAUDE.md (added during this session) now says "Never pipe to a standalone `jq` command" but the skill templates contradict this by providing a copy-paste command with `jq` in it. Agents copy the template verbatim.

### Proposed Fix

Rewrite the Board Movement Helper in all skills to use `gh`'s built-in `--jq` flag:

**Before:**
```bash
gh project item-list 2 --owner luketmoss --format json | jq -r '.items[] | select(.content.number == 29) | .id'
```

**After:**
```bash
gh project item-list 2 --owner luketmoss --format json --jq '.items[] | select(.content.number == 29) | .id'
```

Same change for all 4 nested subshells in the helper (item-list, project list, field-list x2).

### Files to Change
- `.claude/skills/pm/SKILL.md` — Board Movement Helper
- `.claude/skills/dev/SKILL.md` — Board Movement Helper
- `.claude/skills/qa/SKILL.md` — Board Movement Helper
- `.claude/skills/review/SKILL.md` — Board Movement Helper
- `.claude/skills/ux/SKILL.md` — (no board movement helper, but check for ad-hoc jq)
- `.claude/skills/idea/SKILL.md` — (no board movement helper, but check for ad-hoc jq)

## Finding 3: PM Skill Lacks Migration / Day-Zero ACs

**Severity: P1 — Caused 4 post-merge bugs on #41**

### Evidence

From the #41 retrospective, all 4 post-merge bugs:

| Bug | Root Cause | Should Have Been Caught By |
|-----|-----------|---------------------------|
| `fetchBoards` crashes on missing Boards tab | Assumed tab exists | PM (migration AC) |
| No way to create the Boards tab | No bootstrap logic | PM (migration AC) |
| Board switcher invisible with 0 boards | Component guard too aggressive | QA (empty state test) |
| Existing items disappear after first board | No `board_id` on old items | PM (migration AC) + Dev |

The PM's acceptance criteria assumed a greenfield environment. None addressed what happens to existing data/users when the feature deploys.

### Proposed Fix

Add a "Migration & Backward Compatibility" prompt to the PM skill's Step 3 (Write acceptance criteria). When the issue introduces new data structures (tabs, columns, fields), the PM must include:

```markdown
### Migration AC: Existing data remains functional
- **Given** the feature is deployed to an existing installation
- **When** the user loads the app for the first time after the change
- **Then** all existing data remains accessible and functional without manual intervention
```

Also add a checklist item to Step 3:
- Does this feature add new Sheet tabs, columns, or fields? If yes, write migration ACs.

### Files to Change
- `.claude/skills/pm/SKILL.md` — Step 3, Step 6 template, Definition of Done

## Finding 4: QA Tests Only Demo Mode — Misses Empty/Fresh States

**Severity: P1 — Caused 3 of 4 post-merge bugs on #41**

### Evidence

From #41 retrospective: QA tested in `?demo=true` which pre-loads mock boards and items with `board_id` already set. It never tested:
- Zero boards state (Bug 3)
- Items without `board_id` (Bug 4)
- Missing Sheet tabs (Bug 1)

The QA skill has Step 5 (Edge case testing) which lists "Empty states — what happens with no data?" but this is generic guidance. It doesn't specifically instruct QA to test the **bootstrapping/first-time/empty state** for features that add new data models.

### Proposed Fix

Add to QA Step 5:

```markdown
- **Fresh install / empty state** — if this feature introduces a new data model (tab, entity type, field),
  test with zero instances. Verify the UI handles the empty state and the bootstrapping flow works.
  If demo mode masks this (pre-loaded mock data), note it as an untested risk in the report.
```

### Files to Change
- `.claude/skills/qa/SKILL.md` — Step 5

## Finding 5: Dev Skill Lacks Backward-Compatibility Awareness

**Severity: P1 — Contributed to #41 bugs**

### Evidence

The Dev skill has a "Rules Sync" check (Step 5) and a PR template with a "Rules Sync" section, but nothing about backward compatibility with existing data.

From #41: Dev added `board_id` to the Item model and filtered by it, but never wrote migration logic for existing items that lack `board_id`.

### Proposed Fix

Add a checklist item to Dev's Step 4 (Implement with tests):

```markdown

- What happens to existing rows/items that don't have this field?
- Does the UI handle the zero-instances case?
- Is there migration or bootstrapping logic needed?
```

## Finding 6: Redundant Build Verification Between QA and Review

**Severity: P2 — ~20 seconds wasted per pipeline run**

### Evidence

From the #77 analysis and the 3-issue batch analysis:

Both QA (Step 2) and Review (Section 6) run the identical verification:
```bash
cd frontend && npm test
cd apps-script && npm test
cd frontend && npx tsc --noEmit
cd frontend && npm run build
```

The Review agent re-runs everything QA just verified on the same commit.

### Proposed Fix

Add a note to Review's Section 6 (Build Verification):

```markdown
If QA has already passed on this branch (check for a PASS QA report on the PR),
you may skip the full build verification and instead just run `npm test` as a sanity check.
```

### Files to Change
- `.claude/skills/review/SKILL.md` — Section 6

## Finding 7: Redundant Issue Reads Across Skills

**Severity: P2 — ~5K tokens wasted per pipeline run**

### Evidence

From #77 analysis: the issue was read via `gh issue view` **6 times** — once by each skill (PM, UX, PM+UX negotiation, Dev, QA, Review). Each read returns the full issue body including all ACs, scope, and technical notes.

### Proposed Fix

Update CLAUDE.md pipeline instructions. The orchestrator should:
1. Read the issue once at pipeline start
2. Pass relevant context (ACs, scope, technical notes) as part of the skill invocation prompt
3. Skills still read the issue for their own verification, but the orchestrator provides a summary to reduce the need

Add to CLAUDE.md Pipeline Orchestration preamble:

```markdown
**Context passing:** Read the issue once at the start of the pipeline. When invoking each skill,
include the current ACs and scope in your prompt so the skill has immediate context.
Skills may still read the issue for verification, but this reduces redundant API calls.
```

### Files to Change
- `CLAUDE.md` — Pipeline Orchestration preamble

## Finding 8: QA Uses Excessive Visual Testing

**Severity: P2 — 34K tokens, 24.7% of active time**

### Evidence

From #77 analysis: QA made 42 tool calls (30 preview tools), consuming +34K tokens — the single largest context consumer. It took screenshots at 3 viewports, inspected multiple elements, clicked through interactions, and resized repeatedly.

For ACs that are purely behavioral/data-driven (not visual), the unit tests already verify correctness. Visual testing should focus on UI-specific ACs.

### Proposed Fix

Add guidance to QA Step 4 (Visual and functional testing):

```markdown
**Focus visual testing on UI-specific ACs.** If an AC is about logic, data, or behavior
(e.g., "items are filtered by X"), verify via unit tests and code review in Steps 2-3.
Reserve visual/preview testing for ACs about appearance, layout, interaction, and responsive behavior.
This reduces context usage and speeds up the QA stage.
```

### Files to Change
- `.claude/skills/qa/SKILL.md` — Step 4

## Finding 11: Retrospective Skill for Continuous Improvement

**Severity: P1 — Enables continuous improvement of the pipeline itself**

### Problem

Pipeline improvements to date have come from manual, ad-hoc retrospectives. These are high-value (every improvement in this audit traces back to one) but inconsistent — they only happen when someone remembers to do one. Meanwhile, issues repeat across sessions because lessons aren't captured systematically.

### Design

A `/retro` skill that the user invokes **at the end of a pipeline session, in the same conversation**. Because it runs in-session, it has full access to the conversation context — every skill invocation, every error, every user nudge. GitHub artifacts are supplementary verification, not the primary data source.

**Input:** Issue numbers worked on this session (e.g., `/retro 29`)

**Primary data source — conversation context:**
- Which skills were invoked and in what order
- What each skill returned
- Errors and command failures
- User interventions and course-corrections
- Whether the orchestrator stalled between stages

**Supplementary — GitHub artifacts (for verification):**
- Issue comments (were agent reports actually posted?)
- PR timeline and review status
- Board state (is the issue in the right column?)

**Output:** A markdown report in `.claude/retrospectives/` with:
- Pipeline compliance table (which stages ran, output quality)
- Findings categorized as STALL, SKIPPED, INCOMPLETE, DEVIATION, QUALITY GAP, EFFICIENCY, ERROR, or PATTERN
- Cross-reference against previous retros for recurring patterns
- Suggested fixes with specific file paths and proposed text

**Workflow:**
1. User finishes a pipeline session and runs `/retro 29`
2. Skill analyzes the conversation context + verifies GitHub artifacts
3. Writes report to `.claude/retrospectives/retro-issue-<N>-<date>.md`
4. Periodically, user reviews accumulated retros and implements changes
5. Reviewed retros move to `.claude/retrospectives/Reviewed/`

**What the skill should NOT do:**
- Auto-edit skill files or CLAUDE.md (changes need human review)
- Run mid-pipeline (invoke it at the end, after the work is done)
- Produce lengthy documents when there are no findings (short "clean run" reports are fine)

### Files to Create
- `.claude/skills/retro/SKILL.md`
