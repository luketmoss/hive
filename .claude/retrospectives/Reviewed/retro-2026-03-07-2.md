## New Findings

### Finding N1: Context Compaction Mid-Pipeline Is Now the Primary Risk

- **Type:** EFFICIENCY
- **Severity:** P1
- **What happened:** The session hit context compaction mid-QA, requiring session resumption. The orchestrator recovered cleanly from the conversation summary, but some nuance from earlier stages was lost (e.g., exact PM negotiation reasoning, dev implementation decisions).
- **Expected behavior:** CLAUDE.md says "Steps 1-5 are autonomous." The pipeline is designed for single-session execution. Context compaction breaks this assumption.
- **Root cause:** With the idle stalls fixed (Finding 1), the pipeline now runs efficiently — but a full 7-stage pipeline (PM, UX, Negotiation, Dev, QA, Review, Merge) with deep codebase exploration genuinely consumes near-maximum context.
- **Suggested fix:** This is an inherent constraint, not a skill/prompt issue. Options:
  1. Accept that full pipeline runs may require one compaction event. The recovery path works.
  2. For future optimization, consider lighter-weight skill invocations that don't re-read the entire codebase (Dev and QA are the heaviest consumers).
  3. The `pipeline-analysis-2026-03-06.md` previously suggested per-stage session boundaries — this would help but adds complexity.

### Finding N2: Orchestrator Ran QA and Review Directly After Compaction

- **Type:** DEVIATION
- **Severity:** P2
- **What happened:** After context compaction, the orchestrator completed QA verification itself (running tests, checking the preview, posting the QA report) rather than invoking the `/qa` skill. It then invoked `/review` as a skill correctly.
- **Expected behavior:** CLAUDE.md says "Invoke `/qa` with the issue number." The QA should run as a skill invocation, not as inline orchestrator work.
- **Impact:** Low — the QA work was thorough and correct. The report was posted in the expected format. But running QA inline means the orchestrator consumed more context on QA work than a delegated skill would.
- **Suggested fix:** After context compaction, the orchestrator should still invoke skills rather than doing the work inline. The conversation summary provides enough context for skill invocation. No file changes needed — this is model behavior during recovery.

### Finding N3: `devops` and `orchestrator` Skills Still Have `| jq`

- **Type:** INCOMPLETE
- **Severity:** P2
- **What happened:** Finding 2 from the optimization audit was applied to the 4 core pipeline skills (pm, dev, qa, review) but missed `devops/SKILL.md` and `orchestrator/SKILL.md`, which still use `| jq` pipe syntax.
- **Expected behavior:** The audit said "Same change for all 4 nested subshells in the helper" and listed 6 files to check, but `devops` and `orchestrator` were listed with "check for ad-hoc jq" which was apparently not done.
- **Suggested fix:**
  - `.claude/skills/devops/SKILL.md` — Replace `| jq -r` with `--jq` in Board Movement Helper (lines 135-138)
  - `.claude/skills/orchestrator/SKILL.md` — Replace `| jq -r` with `--jq` in Board Movement Helper (lines 30-33) and status check (line 79)
